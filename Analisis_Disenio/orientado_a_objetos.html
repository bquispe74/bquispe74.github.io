Orientado a Objetos

La programación orientada a objetos (POO) es un paradigma que organiza el software en torno a clases y objetos, es decir, unidades que combinan datos y comportamiento. Sus pilares principales son: encapsulamiento, herencia, polimorfismo y abstracción.

Estos principios permiten construir sistemas más modulares, reutilizables y fáciles de mantener. En el caso del sistema de Patata Republic, se utilizan clases para representar actores del negocio (cliente, cajero, cocinero, administrador), pedidos, productos, inventario, entre otros elementos.

Para documentar y diseñar este tipo de sistemas se utiliza UML (Unified Modeling Language), un lenguaje de modelado visual que define diferentes tipos de diagramas estructurales y de comportamiento. A continuación se presentan algunos de los diagramas UML más relevantes del proyecto.

Diagrama UML general

El diagrama UML general muestra la estructura estática del sistema: las principales clases del dominio, sus atributos, operaciones y las relaciones entre ellas (asociaciones, agregaciones, composiciones y herencias).

Este diagrama sirve como mapa global del sistema de Patata Republic, ayudando a entender cómo se organizan las responsabilidades de cada clase y cómo colaboran para cubrir los procesos de venta, preparación de pedidos y gestión de inventario.

Diagrama de casos de uso de negocio

El diagrama de casos de uso de negocio describe los procesos principales de la organización desde el punto de vista del negocio, antes de pensar en la parte tecnológica. Se identifican actores como el cliente, el cajero o el proveedor, y se muestra qué servicios ofrece Patata Republic para cada uno de ellos.

Este modelo ayuda a comprender qué hace la empresa y cómo se relaciona con su entorno: realizar pedidos, preparar alimentos, entregar productos, gestionar insumos, etc. Es la base para decidir qué funcionalidades deben ser apoyadas luego por el sistema informático.

Diagrama de casos de uso del sistema

El diagrama de casos de uso del sistema detalla cómo los actores interactúan con el software de Patata Republic. Cada caso de uso representa una funcionalidad observada por el usuario, como registrar un pedido, actualizar el menú, controlar el stock de papas o generar reportes de ventas.

A partir de estos casos de uso se definen los requisitos funcionales, se elaboran escenarios y se diseñan pruebas. Es un puente entre el análisis del negocio y el diseño orientado a objetos del sistema.

Caso de uso de tipo texto expandido

6.3 Caso de uso de tipo texto expandido

CASO DE USO: Realizar pedido de papas fritas (tipo texto expandido)

Código: CU-01

ACTORES

  • Cliente (iniciador)
  • Vendedor/Cajero (actor principal)
  • Operario de cocina (actor interesado)
  • Administrador (actor interesado en reportes e inventario, de forma indirecta)

PROPÓSITO

Registrar un pedido de papas fritas y bebidas solicitado por un Cliente, calculando el total a pagar, generando la orden para cocina y dejando el pedido listo para el registro de pago, la preparación y la entrega.

RESUMÉN

Un Cliente acude al local de “Patata Republic” y solicita un pedido de papas fritas y/o bebidas. El Vendedor/Cajero abre la opción “Realizar pedido” en el sistema, consulta el menú y precios (CU2: Consultar menú y precios), registra los productos solicitados por el Cliente y el sistema calcula el total del pedido. Una vez que el Cliente confirma, el Vendedor/Cajero registra el pago (CU3: Registrar pago del pedido), el sistema genera el comprobante y crea la orden de preparación para cocina (CU4: Preparar pedido en cocina). Cuando el Operario de cocina finaliza la preparación, el pedido queda listo para ser entregado al Cliente (CU5: Entregar pedido al cliente).

TIPO

Primario y esencial.

REFERENCIAS CRUZADAS

  • CU2: Consultar menú y precios (include)
  • CU3: Registrar pago del pedido (include)
  • CU4: Preparar pedido en cocina (include)
  • CU5: Entregar pedido al cliente (include)
  • Requisitos: R1.1 (Gestión de pedidos), R1.2 (Cálculo de totales), R1.3 (Integración con pago), R1.4 (Orden para cocina)

PRECONDICIONES

  • El sistema de ventas está operativo.
  • El Vendedor/Cajero está autenticado en el sistema.
  • El catálogo de productos y precios (menú) está configurado y vigente.
  • Opcional: El inventario de insumos está actualizado (para verificar disponibilidad).

POSTCONDICIONES

  • El pedido del Cliente queda registrado en el sistema con su estado correspondiente (por ejemplo: “Pagado” y “En preparación” o según política del negocio).
  • El detalle del pedido (productos, cantidades, precios, total) queda almacenado para futuras consultas y reportes de ventas.
  • Se genera y asocia una orden de preparación para cocina.
  • En caso de cancelación antes del pago, no se genera pedido definitivo ni se afectan inventarios.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Cliente se acerca al mostrador y solicita al Vendedor/Cajero realizar un pedido de papas fritas y/o bebidas. 2. Muestra la opción “Realizar pedido” en el módulo de ventas.
El Vendedor/Cajero selecciona la opción “Realizar pedido”. 4. Abre la pantalla de nuevo pedido y muestra el menú de productos y precios (incluye CU2: Consultar menú y precios).
El Vendedor/Cajero consulta junto al Cliente las opciones de papas fritas, tamaños, acompañamientos y bebidas, y selecciona los productos deseados. 6. Registra en el detalle del pedido los productos seleccionados, cantidades y precios, y muestra el subtotal actualizado.
El Vendedor/Cajero confirma con el Cliente que el detalle del pedido es correcto (productos, tamaños, salsas, bebidas). 8. Recalcula el total del pedido (subtotal, impuestos si aplica, descuentos) y muestra el monto final a pagar.
El Cliente acepta el pedido y el Vendedor/Cajero procede a registrar el pago. 10. Abre la opción de “Registrar pago” (incluye CU3: Registrar pago del pedido) y solicita el tipo de pago (efectivo, tarjeta, QR, etc.).
El Vendedor/Cajero ingresa el tipo de pago y el monto recibido. 12. Valida el pago, registra el comprobante y marca el pedido como “Pagado”.
Una vez registrado el pago, el Vendedor/Cajero confirma la generación de la orden para cocina. 14. Genera la orden de preparación para el Operario de cocina (incluye CU4: Preparar pedido en cocina), asignando un número de pedido y el detalle a preparar.
El Operario de cocina visualiza la orden y comienza la preparación del pedido. 16. Actualiza el estado del pedido a “En preparación” y permite marcarlo como “Listo” cuando el Operario de cocina termine.
Al estar listo, el Operario de cocina marca el pedido como “Listo para entrega”. 18. Actualiza el estado del pedido a “Listo” y notifica en la interfaz del Vendedor/Cajero.
El Vendedor/Cajero llama al Cliente y realiza la entrega del pedido. 20. Registra la entrega (incluye CU5: Entregar pedido al cliente) y cambia el estado del pedido a “Entregado”, finalizando la atención.

CURSOS ALTERNOS

Línea 4: Menú o productos no disponibles o desactualizados

  • 4a. El sistema indica que faltan productos configurados o que algunos están inactivos.
  • 4b. El Vendedor/Cajero informa al Cliente de la situación y ofrece alternativas (otros productos o tamaños).
  • 4c. Si el Cliente no acepta ninguna alternativa, el Vendedor/Cajero cancela la operación y el caso de uso termina sin registrar pedido.

Línea 5–6: Producto sin stock suficiente

  • 6a. El sistema detecta que el stock de cierto producto o insumo es insuficiente (por integración con CU7).
  • 6b. Muestra mensaje “Stock insuficiente para [producto]”.
  • 6c. El Vendedor/Cajero puede reducir la cantidad del producto, sustituirlo por otro o eliminarlo del pedido.
  • 6d. Una vez ajustado el detalle, el sistema recalcula el total y se regresa a la línea 7 de la secuencia normal.

Línea 8: El Cliente no está de acuerdo con el total

  • 8a. El Cliente solicita cambiar algún producto o tamaño.
  • 8b. El Vendedor/Cajero modifica el detalle del pedido (agrega, quita o cambia productos).
  • 8c. El sistema recalcula el total y vuelve a mostrar el monto final.
  • 8d. Se regresa a la línea 9 de la secuencia normal.

Línea 10–12: Problema con el pago

  • 12a. En pago con tarjeta/QR, el sistema o el medio de pago indica error o transacción rechazada.
  • 12b. El sistema muestra mensaje “Pago no autorizado” o “Error en la transacción”.
  • 12c. El Vendedor/Cajero ofrece al Cliente usar otro medio de pago o cancelar el pedido.
  • 12d. Si el Cliente realiza otro intento de pago, se repite el paso 11 y el sistema intenta nuevamente registrar el pago.
  • 12e. Si el Cliente decide no continuar, el Vendedor/Cajero cancela el pedido y el caso de uso termina sin registrar venta.

Línea 14–16: Sobrecarga en cocina o error de envío de orden

  • 16a. El sistema no logra enviar la orden a la pantalla de cocina (error de conexión).
  • 16b. Muestra mensaje de error al Vendedor/Cajero: “No se pudo enviar la orden a cocina”.
  • 16c. El Vendedor/Cajero puede reintentar el envío o, si el problema persiste, imprimir la orden en papel como respaldo.
  • 16d. Una vez enviada o registrada manualmente, se retoma la secuencia normal en la línea 15.

Línea 19: Cliente no aparece para recoger el pedido (para pedidos adelantados)

  • 19a. El tiempo de espera máximo se cumple y el Cliente no se presenta.
  • 19b. El sistema permite marcar el pedido como “No recogido” o “Anulado por inasistencia” según política del negocio.
  • 19c. El Administrador puede revisar posteriormente estos casos en reportes de ventas y mermas.

CASO DE USO: Consultar menú y precios

Código: CU-02

ACTORES

  • Cliente (actor iniciador)
  • Vendedor/Cajero (actor principal)

PROPÓSITO

Permitir la visualización del menú de productos disponibles (papas fritas, tamaños, combos, bebidas y acompañamientos) junto con sus precios vigentes, para apoyar la decisión de compra del Cliente y facilitar el registro correcto del pedido.

RESUMÉN

Un Cliente se acerca al local y desea conocer las opciones de papas fritas, bebidas y sus precios antes de realizar un pedido. El Vendedor/Cajero accede al módulo “Consultar menú y precios”, donde el sistema muestra el listado de productos activos, sus descripciones, tamaños disponibles, combinaciones (combos) y precios actualizados. Esta información puede ser utilizada directamente para informar al Cliente o para continuar con el registro del pedido (CU-01: Realizar pedido de papas fritas).

TIPO

Secundario (de apoyo) y esencial.

REFERENCIAS CRUZADAS

  • CU-01: Realizar pedido de papas fritas (lo incluye)
  • CU-06: Gestionar productos y precios
  • Requisitos: R2.1 (Visualización de menú), R2.2 (Precios vigentes), R2.3 (Productos activos)

PRECONDICIONES

  • El sistema de ventas está operativo.
  • El catálogo de productos ha sido previamente configurado por el Administrador.
  • El Vendedor/Cajero está autenticado en el sistema (si se accede desde terminal interna).

POSTCONDICIONES

  • No se generan cambios en la base de datos.
  • El menú y los precios mostrados pueden ser utilizados para iniciar o continuar un pedido de venta.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Cliente solicita información sobre los productos y precios disponibles. 2. Muestra la opción “Consultar menú y precios”.
El Vendedor/Cajero selecciona “Consultar menú y precios”. 4. Recupera del sistema el listado de productos activos.
El Vendedor/Cajero solicita ver tipos de papas fritas y sus tamaños. 6. Muestra las variedades de papas fritas, tamaños disponibles y precios correspondientes.
El Cliente pregunta por acompañamientos, salsas o bebidas. 8. Muestra acompañamientos y bebidas disponibles con sus precios.
El Vendedor/Cajero revisa si existen combos u ofertas vigentes. 10. Muestra combos activos, promociones y precios especiales, si existen.
El Cliente decide qué productos desea pedir. 12. Mantiene visible la información del menú para continuar con el registro del pedido o permite salir del módulo.

CURSOS ALTERNOS

Línea 4: No existen productos activos

  • 4a. El sistema muestra el mensaje “No hay productos disponibles en el menú”.
  • 4b. El Vendedor/Cajero informa al Cliente y recomienda esperar o regresar más tarde.
  • 4c. El caso de uso finaliza.

Línea 6: Algunos productos no están disponibles por falta de stock

  • 6a. El sistema indica visualmente qué productos están sin stock o temporalmente no disponibles.
  • 6b. El Vendedor/Cajero informa al Cliente y sugiere productos alternativos.

Línea 8: Precios en actualización

  • 8a. El sistema detecta que un producto tiene precio pendiente de actualización.
  • 8b. Muestra mensaje “Precio en actualización, consultar con administrador”.
  • 8c. El Vendedor/Cajero evita ofrecer ese producto o informa al Cliente del precio referencial.

CASO DE USO: Realizar pedido de papas fritas

Código: CU-01

ACTORES

  • Cliente (iniciador)
  • Vendedor/Cajero (actor principal)

PROPÓSITO

Registrar el pedido de papas fritas, bebidas y otros productos solicitados por el Cliente, calcular el monto total y dejar el pedido listo para el cobro, preparación y entrega.

RESUMÉN

El Cliente se acerca al mostrador y solicita un pedido de papas fritas y/o bebidas. El Vendedor/Cajero consulta el menú y precios disponibles (incluye CU-02), registra los productos seleccionados, cantidades y observaciones. El sistema calcula el total del pedido, muestra el detalle y permite confirmar la orden. Una vez confirmado, el caso de uso incluye el proceso de registrar el pago (CU-03), enviar el pedido a cocina para su preparación (CU-04) y posteriormente su entrega al Cliente (CU-05). El pedido queda almacenado con un identificador único y su estado inicial (por ejemplo: “pendiente de pago” o “en preparación”, según el flujo definido).

TIPO: Primario y esencial

REFERENCIAS CRUZADAS: R1.1, R1.2, R1.3, CU-02, CU-03, CU-04, CU-05

PRECONDICIONES

  • El sistema de ventas está operativo.
  • El Vendedor/Cajero está autenticado en el sistema.
  • El catálogo de productos y precios está configurado (CU-06).

POSTCONDICIONES

  • El pedido queda registrado con su detalle de productos, cantidades y total.
  • Queda disponible para el proceso de pago, preparación en cocina y entrega.
  • Si el Cliente cancela antes de confirmar, no se registra el pedido.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Cliente solicita al Vendedor hacer un pedido de papas fritas y/o bebidas. 2. Muestra la opción “Registrar pedido” y el menú de productos (incluye CU-02).
El Vendedor selecciona “Registrar pedido”. 4. Presenta la interfaz de captura del pedido (lista de productos, tamaños, combos, bebidas).
El Vendedor selecciona los productos, cantidades y registra observaciones (salsas, tipo de corte, etc.). 6. Calcula subtotales y el total del pedido en tiempo real.
El Vendedor confirma el detalle del pedido con el Cliente. 8. Muestra el resumen del pedido (productos, cantidades, total) y solicita confirmación final.
El Vendedor confirma el pedido. 10. Registra el pedido en la base de datos, genera un identificador de pedido y lo marca como “pendiente de pago” o según la política del local.
El Vendedor procede al registro del pago. 12. Invoca CU-03: Registrar pago del pedido.
(Tras el pago) El Vendedor envía el pedido a cocina. 14. Actualiza el estado del pedido y lo envía al módulo de cocina (incluye CU-04).

CURSOS ALTERNOS

Línea 6: Falta de stock de algún producto.

  • 6a. El sistema indica que cierto producto o tamaño no tiene stock suficiente.
  • 6b. El Vendedor propone un producto alternativo o modifica la cantidad.
  • 6c. Se recalcula el total y se regresa a la línea 7 de la secuencia normal.

Línea 9: El Cliente decide cancelar el pedido.

  • 9a. El Vendedor cancela la operación.
  • 9b. El sistema descarta la información temporal y el caso de uso termina sin registrar el pedido.

Línea 12 (CU-03): El pago es rechazado.

  • 12a. El sistema indica “Pago no aprobado” o “Error en el cobro”.
  • 12b. El Vendedor puede intentar nuevamente el pago o cancelar el pedido.
  • 12c. Si cancela, el pedido queda anulado o marcado como “cancelado”.

ACTORES

Línea 2: No hay productos configurados.

  • 2a. El sistema muestra mensaje “No hay productos disponibles” o “Catálogo vacío”.

CASO DE USO: Registrar pago del pedido

Código: CU-03

ACTORES

  • Cliente (actor interesado)
  • Vendedor/Cajero (actor principal)

PROPÓSITO

Registrar el cobro de un pedido, indicando el monto pagado y la forma de pago, y generar el comprobante correspondiente.

RESUMÉN

Una vez confirmado el pedido, el Vendedor/Cajero solicita al Cliente el pago. El sistema muestra el total a pagar y permite seleccionar la forma de pago (efectivo, tarjeta, QR, etc.). El Vendedor registra el monto recibido y el sistema calcula el cambio, si corresponde. El sistema registra la transacción, asocia el pago al pedido y genera un comprobante de venta. Este caso de uso normalmente es incluido por CU-01.

TIPO: Primario y esencial

REFERENCIAS CRUZADAS: R3.1, R3.2, R3.3, CU-01, CU-09

PRECONDICIONES

  • Existe un pedido registrado y confirmado (CU-01).
  • El Vendedor/Cajero está autenticado en el sistema.

POSTCONDICIONES

  • El pago queda registrado y asociado al pedido.
  • El pedido cambia de estado (por ejemplo, a “pagado/en preparación”).
  • Se genera un comprobante de pago/venta.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Vendedor selecciona la opción “Registrar pago” para un pedido. 2. Muestra el detalle del pedido y el total a pagar.
El Vendedor indica la forma de pago (efectivo, tarjeta, QR, etc.). 4. Valida la forma de pago y habilita los campos necesarios (por ejemplo, número de voucher).
El Cliente entrega el pago o autoriza la transacción. 6. Registra el monto recibido y, si es efectivo, calcula el cambio.
El Vendedor confirma el registro del pago. 8. Registra la transacción en la base de datos, asocia el pago al pedido y actualiza el estado.
El sistema genera el comprobante. 10. Muestra y/o imprime el comprobante de pago para el Cliente.

CURSOS ALTERNOS

Línea 4–6: Fallo en el pago electrónico (tarjeta/QR).

  • 6a. El sistema informa “Transacción rechazada” o “Error en el servicio de pago”.
  • 6b. El Vendedor ofrece otro método de pago o cancelar el pedido.
  • 6c. Si el Cliente elige otro método, se regresa a la línea 3.
  • 6d. Si se cancela, el pedido se marca como “cancelado” según la política del local.

CASO DE USO: Preparar pedido en cocina

Código: CU-04

ACTORES

  • Operario de cocina (actor principal)
  • Vendedor/Cajero (actor interesado)

PROPÓSITO

Gestionar la preparación de los pedidos en la cocina, permitiendo al Operario visualizar, preparar y actualizar el estado de cada pedido.

RESUMÉN

Los pedidos confirmados y pagados se envían al módulo de cocina. El Operario de cocina consulta la lista de pedidos pendientes, selecciona uno y prepara las papas fritas y otros productos solicitados. El sistema permite marcar el pedido como “en preparación” y luego “listo”, notificando al Vendedor/Cajero para la entrega al Cliente.

TIPO: Primario y esencial

REFERENCIAS CRUZADAS: R4.1, R4.2, CU-01, CU-05, CU-07

PRECONDICIONES

  • El pedido fue registrado y, según política, pagado (CU-01 y CU-03).
  • El Operario de cocina está autenticado en el sistema (si aplica).

POSTCONDICIONES

  • El pedido cambia de estado a “listo para entregar”.
  • Se actualiza el tiempo de preparación (si el sistema lo registra).

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Operario de cocina accede al módulo “Pedidos en cocina”. 2. Muestra la lista de pedidos pendientes con su detalle básico.
El Operario selecciona un pedido a preparar. 4. Muestra el detalle del pedido (productos, tamaños, observaciones).
El Operario inicia la preparación. 6. Actualiza el estado del pedido a “en preparación”.
El Operario finaliza la preparación del pedido. 8. Cambia el estado a “listo” y notifica al módulo de caja/venta.

CURSOS ALTERNOS

Línea 2: No hay pedidos pendientes.

  • 2a. El sistema muestra mensaje “No hay pedidos en cocina”.

Línea 5: Falta alguno de los insumos (papas, aceite, envases, etc.).

  • 5a. El Operario registra que falta un insumo.
  • 5b. El sistema registra la incidencia y puede disminuir stock (CU-07) o marcar insumo crítico.
  • 5c. El Operario informa al Vendedor para reprogramar o modificar el pedido.

CASO DE USO: Entregar pedido al cliente

Código: CU-05

ACTORES

  • Cliente (actor interesado)
  • Vendedor/Cajero (actor principal)

PROPÓSITO

Registrar que el pedido preparado fue efectivamente entregado al Cliente y cerrar la atención.

RESUMÉN

Una vez que el pedido está marcado como “listo” en cocina, el Vendedor/Cajero llama al Cliente, verifica el número de pedido o nombre y le entrega el pedido. El sistema registra la entrega y cambia el estado del pedido a “entregado” o “finalizado”.

TIPO: Primario y esencial

REFERENCIAS CRUZADAS: R5.1, CU-01, CU-03, CU-04

PRECONDICIONES

  • El pedido está preparado y marcado como “listo” (CU-04).
  • El pago ha sido registrado, si la política es pago previo (CU-03).

POSTCONDICIONES

  • El pedido queda marcado como “entregado” o “finalizado”.
  • Puede utilizarse la información para estadísticas o reportes de ventas (CU-09).

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Vendedor verifica en el sistema los pedidos “listos para entregar”. 2. Muestra la lista de pedidos listos con su identificador y nombre del Cliente.
El Cliente se presenta para recoger su pedido. 4. Permite al Vendedor buscar el pedido por número o nombre.
El Vendedor confirma la identidad del Cliente y la correspondencia del pedido. 6. Muestra el detalle del pedido seleccionado.
El Vendedor entrega el pedido al Cliente. 8. Cambia el estado del pedido a “entregado/finalizado” y registra la hora de entrega.

CURSOS ALTERNOS

Línea 3: El Cliente no se presenta durante un tiempo razonable.

  • 3a. El Vendedor puede marcar el pedido como “no recogido” según política.
  • 3b. El sistema registra el estado para control interno.

CASO DE USO: Gestionar productos y precios

Código: CU-06

ACTORES

  • Administrador (actor principal)

PROPÓSITO

Permitir al Administrador crear, modificar, desactivar productos del menú y actualizar sus precios.

RESUMÉN

El Administrador accede al módulo de “Gestión de productos y precios”. Puede registrar nuevos productos (por ejemplo, nuevos tipos de papas fritas, combos, bebidas), editar la descripción, categoría, tamaño y precio, o desactivar productos que ya no se venden. El sistema valida la información, guarda los cambios y actualiza el catálogo utilizado en ventas y consultas de menú (CU-01, CU-02).

TIPO: Primario (administrativo) y esencial

REFERENCIAS CRUZADAS: R6.1, R6.2, CU-01, CU-02, CU-09

PRECONDICIONES

  • El Administrador está autenticado con permisos de gestión.

POSTCONDICIONES

  • Los productos y precios quedan actualizados en el catálogo.
  • Los cambios impactan en las futuras ventas y consultas de menú.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Administrador accede a “Gestionar productos y precios”. 2. Muestra la lista de productos actuales con sus precios y estados (activo/inactivo).
El Administrador selecciona “Crear producto” o “Editar producto”. 4. Presenta el formulario para registrar o actualizar datos (nombre, categoría, precio, etc.).
El Administrador ingresa o modifica la información del producto. 6. Valida que los campos obligatorios estén completos y el formato sea correcto.
El Administrador confirma la acción (crear/actualizar/desactivar). 7. Registra los cambios en la base de datos y actualiza el catálogo.

CURSOS ALTERNOS

Línea 6: Datos incompletos o inválidos.

  • 6a. El sistema indica qué campos están incorrectos o vacíos.
  • 6b. El Administrador corrige la información.
  • 6c. Regresa a la línea 7 de la secuencia normal.

CASO DE USO: Gestionar inventario de insumos

Código: CU-07

ACTORES

  • Operario de cocina (actor interesado)
  • Administrador (actor principal)

PROPÓSITO

Registrar y actualizar el stock de insumos (papas, aceite, salsas, envases, etc.), identificar insumos críticos y controlar los movimientos de inventario.

RESUMÉN

El Administrador o persona encargada de inventario accede al módulo de “Inventario de insumos”. Puede revisar el stock actual, registrar ajustes manuales (mermas, correcciones) y consultar insumos con stock crítico. Este caso de uso incluye el registro de recepción de insumos de proveedores (CU-08) cuando se ingresan nuevas compras.

TIPO: Primario (administrativo) y esencial

REFERENCIAS CRUZADAS: R7.1, R7.2, CU-04, CU-08, CU-10

PRECONDICIONES

  • El Administrador está autenticado.
  • Existe un catálogo de insumos configurado.

POSTCONDICIONES

  • El inventario de insumos se actualiza.
  • Se registran los movimientos realizados (ingresos, egresos, ajustes).

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Administrador accede a “Gestionar inventario de insumos”. 2. Muestra el listado de insumos con stock actual y estado (normal/crítico).
El Administrador selecciona un insumo para revisar o ajustar. 4. Muestra el detalle del insumo y su historial de movimientos.
El Administrador registra un ajuste de stock (por ejemplo, merma o corrección). 6. Valida los datos del ajuste (cantidad, motivo).
El Administrador confirma el ajuste. 8. Registra el movimiento en la base de datos y actualiza el stock.
El Administrador registra una nueva entrada de insumos. 10. Invoca CU-08: Registrar recepción de insumos del proveedor.

CURSOS ALTERNOS

Línea 2: No hay insumos registrados.

  • 2a. El sistema muestra mensaje “No hay insumos configurados”.
  • 2b. El Administrador debe registrar insumos iniciales (fuera del alcance o mediante otro CU).

CASO DE USO: Registrar recepción de insumos del proveedor

Código: CU-08

ACTORES

  • Proveedor (actor interesado)
  • Administrador (actor principal)

PROPÓSITO

Registrar las compras de insumos a proveedores y el ingreso de dichos insumos al inventario.

RESUMÉN

Cuando llega un pedido de insumos (papas, aceite, salsas, envases, etc.), el Administrador registra la recepción: selecciona el Proveedor, indica los insumos recibidos, cantidades, precios de compra y fecha. El sistema valida los datos, actualiza el inventario de insumos (CU-07) y registra el movimiento de ingreso para fines de control y reporte.

TIPO: Primario (administrativo) y esencial

REFERENCIAS CRUZADAS: R8.1, R8.2, CU-07, CU-10

PRECONDICIONES

  • El Proveedor está registrado en el sistema.
  • El Administrador está autenticado.

POSTCONDICIONES

  • Se registra una compra/recepción de insumos.
  • El stock de los insumos se incrementa según las cantidades recibidas.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Administrador accede a “Registrar recepción de insumos”. 2. Muestra formulario para registrar proveedor, fecha y detalle de insumos.
El Administrador selecciona al Proveedor. 4. Valida que el proveedor exista y muestra sus datos básicos.
El Administrador ingresa los insumos recibidos (tipo, cantidad, precio unitario). 6. Valida formato de datos (cantidades numéricas, precios válidos).
El Administrador confirma la recepción. 8. Registra la compra en la base de datos, actualiza stock de cada insumo y genera un registro de movimiento de inventario.

CURSOS ALTERNOS

Línea 4: El proveedor no está registrado.

  • 4a. El sistema indica “Proveedor no encontrado”.
  • 4b. El Administrador puede registrar un nuevo proveedor (si existe un CU correspondiente) o cancelar la operación.

CASO DE USO: Generar reporte de ventas

Código: CU-09

ACTORES

  • Vendedor/Cajero (actor interesado)
  • Administrador (actor principal)

PROPÓSITO

Generar reportes de ventas diarias, semanales o mensuales, por producto, monto total, forma de pago y otros criterios.

RESUMÉN

El Administrador o Vendedor/Cajero con permisos de consulta accede a “Reporte de ventas”. Selecciona un rango de fechas y filtros (por producto, forma de pago, turno, etc.). El sistema procesa la información de los pedidos y pagos registrados, genera un reporte con totales y subtotales y, opcionalmente, incluye información de inventario (CU-10) cuando se requiere un reporte integral de ventas e inventario.

TIPO: Primario (de reporte) y esencial

REFERENCIAS CRUZADAS: R9.1, R9.2, CU-01, CU-03, CU-10

PRECONDICIONES

  • Existen ventas registradas en el sistema.
  • El usuario que genera el reporte está autenticado y tiene permisos.

POSTCONDICIONES

  • Se genera un reporte de ventas en pantalla y/o formato imprimible/exportable.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Administrador accede a “Generar reporte de ventas”. 2. Muestra filtros disponibles (rango de fechas, producto, forma de pago, etc.).
El Administrador selecciona el rango de fechas y filtros deseados. 4. Valida los filtros y prepara la consulta.
El Administrador solicita generar el reporte. 6. Procesa las ventas del período seleccionado y calcula totales y subtotales.
El sistema muestra el reporte de ventas. 8. Presenta la información en tabla y, si se solicita un reporte completo, incluye información de inventario (invoca CU-10).
El Administrador puede exportar o imprimir el reporte. 10. Genera el archivo (por ejemplo, PDF/Excel) o envía a impresión.

CURSOS ALTERNOS

Línea 6: No existen ventas en el período seleccionado.

  • 6a. El sistema muestra mensaje “No se encontraron ventas en el rango seleccionado”.
  • 6b. El Administrador puede cambiar el rango de fechas o cancelar.

CASO DE USO: Generar reporte de inventario

Código: CU-10

ACTORES

  • Administrador (actor principal)

PROPÓSITO

Mostrar la situación del inventario de insumos: stock disponible, insumos críticos y movimientos de entrada y salida en un período determinado.

RESUMÉN

El Administrador accede a “Reporte de inventario” y selecciona si desea ver stock actual, insumos críticos o el historial de movimientos en un rango de fechas. El sistema consulta los registros de inventario (CU-07, CU-08) y presenta un reporte detallado. Este caso de uso puede ser incluido desde CU-09 cuando se requiere un reporte combinado de ventas e inventario.

TIPO: Primario (de reporte) y esencial

REFERENCIAS CRUZADAS: R10.1, R10.2, CU-07, CU-08, CU-09

PRECONDICIONES

  • Existen insumos registrados e inventario previamente gestionado.
  • El Administrador está autenticado.

POSTCONDICIONES

  • Se genera un reporte de inventario en pantalla y/o formato imprimible/exportable.

SECUENCIA NORMAL DE EVENTOS

ACCIONES DEL ACTOR RESPUESTA DEL SISTEMA
El Administrador accede a “Generar reporte de inventario”. 2. Muestra opciones de reporte (stock actual, insumos críticos, movimientos, etc.) y filtros (rango de fechas).
El Administrador selecciona el tipo de reporte e ingresa filtros. 4. Valida los filtros y prepara la consulta.
El Administrador solicita generar el reporte. 5. Consulta los movimientos de inventario y stock actual, y procesa la información.
El sistema muestra el reporte de inventario. 7. Presenta el stock por insumo, marca los insumos críticos y, si se pidió, muestra el historial de movimientos.
El Administrador puede exportar o imprimir el reporte. 9. Genera el archivo y/o envía a impresión.

CURSOS ALTERNOS

Línea 5: No hay movimientos ni datos para el filtro seleccionado.

  • 5a. El sistema muestra “No se encontraron datos de inventario para el rango seleccionado”.
  • 5b. El Administrador puede ajustar filtros o ver el stock general sin filtro por fecha.
